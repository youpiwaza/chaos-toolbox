Chaos toolbox / Nude html

A big but simple bulk of all html tags you usually use in website.
The goal here is to allow you to verify your styles, and also the overall rendering & harmony.
It can also helps to prevent bug and styles conflicts (espceially on updates, it'll help prevent regressions).
And finally this will provide a solid base to create html, providing best practice, optimisations, compatibility and accessibility.

Recommandations
  • Style / If making a big title for a front page or a jumbotron, Use a specific class instead of styling straight h1.
  • SEO / Have a proper title structure, it might be used by some robots for stuff
  • SEO / Avoid using more than one h1 per page
  • SEO / Avoid skipping heading levels: start from h1, then use h2, then h3, etc.
  • SEO / Prefer using links tags inside title, and not the opposite : <h1> <a href="#"> Hey </a> </h1>
  • SEO / Always add a clear descriptive id attribute to your title. This allow link sharing & add to favorite to a specific part of the page, espceially if your page if vertically huge. MDN is a great example.
  • SEO & Accessibility / Some people recommand using more titles, in their context : <section> & <article>, or even in complex <form>s
  • Accessibility / When sectionning, use the aria-labeledby attribute on the parent element to reference the id of the title. Cf. mdn.

Recommandations
  • Style / No matter what, those tags should remain inline
  • Style / When styling your links, you should have 3 colors :
    • One for the initial state
      a, a:active { ... }
    • One for the visited state
      a:visited { ... }
    • One for the user interaction
      a:focus, a:hover { ... }
    • Pay attention to declare those in this specific order, as CSS priority is needed (ex : hover style should display over visited).
      Care if you are used to alphabetical sorting.
  • Compatibility / Even if both the b & i tags are deprecated, you should have basic style for them
  • SEO / You should use strong & em tags to emphasize your contents. It makes reading easier, and also helps robots getting keywords
  • SEO / Links > Always give a title to your links. Preferably diffrent from the title label
  • SEO / Links > Don't forget to use the target attribute. If the link is targeting a resource outside your website, you should go for a target _blank
    If you got a lot of mixed links (internal & external), I'd recommand to add a specific inline icon.

If classic form (not js handled), consider the following attributes :

  • action : target page

Be careful, those attributes can be overided with button or input (usually type button or image) following attribute
  • formaction

If using javascript (or other) method to handle form submission, pay attention to
  • Still use the submit event, in addition with preventDefault js function

      • Pressing the "Enter" key while a field is focused
      • Using tabulations to switch focus to the submit button, then pressing space
      • Other mobile/tablet/accessibility readers shenanigans
  • Pay attention to attach the submit event the form, and not to the submit button
  • Consider using an input type button instead of submit, if you wanna skip preventDefault()

    • Add labels on top of fields
    • Buttons

      • Validation button should be vertically aligned with other fields
      • Regroup buttons
    • Don't use acronyms or programmaing oriented mambo jambo in labels or placeholders

accesskey

You can test quick acces using "Alt" key + shortcut key
Doesn't seems to work with non-focusable elements, like a simple div
  • Allow you to add keyboard shorcuts to quickly navigate to HTML elements
  • Most browsers uses "Alt" + designated shortcut letter, but Firefox uses "Alt" + "Shift" + shortcut key
  • MDN recommandation : Don't add shortcuts infos to the layout, create a separate page with a manual/infos
  • Can't seem to find documentation about a list of available/recommanded shortcuts, but here's what quick tests show
    • Shortcuts aren't case sensitive ("s" = "S")
    • Doesn't seems to work on non-focusable elements, despite been a global attribute
    • Number works ( accesskey="2" ) but it can has erratic behavior
      • Touchpad doesn't seems to work, probably because a conflict with ASCII characters generation
        • Set cursor in text editor, Try keep "Alt" pressed down, type 0201 on your touchpad, let Alt. You're welcome.
      • Still works if using "Alt" + "Shift" + "&", for 1 by example ("Shift" + "&" = 1) ; on pc
    • Special characters, like "&" seems to works
      • I won't recommand using them though, as Firefox is using "Alt" + "Shift" as a shortcut.
      • Surprinsigly enough it will still works, but will add a conflict between different symbols on the same key
      • For example : if I have 2 accesskeys "&" and "1", pressing "Alt" + "Shift" + "&" on firefox will result in always going to the 1 shortcut
    • Multiple characters "bn" won't work
      • Edge will jump for the first letter ; for example in this case it jumps to the element when pressing "Alt" + "b"
    • You can use a space separated list of shortcut characters ; Browsers should use the first working
      • Not tested, and not recommanded by MDN anyway
    • Using multiple occurences of the same shortcut
      • From what i've tested, there's erratic behavior between browsers, won't recommand
      • Chrome jumps to the last (DOM source order, tabindex makes no differences) occurence, only (no matter case, no difference between "y" and "Y"
      • Firefox jumps to the first occurence, then loops between all occurence (DOM source order, tabindex makes no differences)
      • Edge works like chrome, except that using "Alt" + "Shift" + key allow to loops backward ("Alt" + key loops normally).
  • MDN doc : accesskey

tabindex

Click on the first input of each tab and use the tab key multiple times.
No tabindex used, it'll use the DOM order to switch focus when user tabs
SWAG tabindexes used
  • Allow you to specify the order in which the fields will focus when the user uses sequential keyboard navigation (usually tabulation).
    Especially usefull with table, grid or custom (css) displays.
  • Takes integer as values.
    • If the value specified is 0, the order will be specified by the document's source order.
    • Note, if the value specified is negative, the user won't be able to focus to it through this way, but will still be able to focus it another way.
      Recommanded for JS widgets for examples.
    • If several elements shares the same integer, their order relative to each other will be determined by the document's source order.

  • I'd recommand not using it and create properly organized HTML, it can f*ck up accessibility readers
  • MDN tabindex attribute / lots of subtilities I won't list here as I don't recommand using it
    • Especially priority & children block stuff : In those example, when you tab from the first tab last input, you should go to the first of the second tab, but instead it skips all inputs in the second tab.. until the end of the page is reached (no tabindex somewhat equals tabindex 0)
    • Consider using it with autofocus..
  • Note that you can also use Shift + Tab to navigate backward in forms / other ;)

  • Checkboxs & radio buttons are used to propose clear choices to your users
  • You can group inputs by making them share the same name attribute

      If not, no data is sent, not even their name !

    • You can force a mandatory choice by using the required attribute
    • And/or make use of the checked attribute for the radio input
    • For radio buttons, you need to apply required only once, according that all inputs share the same name attribute
    • You can still apply the required attribute to all inputs
    • Source : Stackoverflow : radio required
  • The value attribute is required
  • You can use the checked attribute (without value) in order to make inputs checked by default
    • If multiple radio inputs are checked in a same name space, the last one (DOM source order) is actually checked
